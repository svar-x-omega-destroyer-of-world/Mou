# Mou Execution Plan (Phase-Wise)

Based on the Mou SRS v2.0, here is a comprehensive execution plan designed for a two-person team. The architecture cleanly separates the backend (FastAPI) and the two frontends (Flutter app, Next.js dashboard) via a fixed JSON API contract, which naturally allows for significant parallel development. 

The build window is short (June 14–21, 2026), so the plan prioritizes de-risking the OCR (the biggest technical risk) immediately and working independently against mocked APIs.

---

## Phase 1: Setup, De-risking & Contract Locking (Day 1-2)
**Goal:** Prove the highest risk (OCR) early, establish repositories, lock the API contract, and setup unblocking mocks.

| Team Member 1 (Focus: Backend & AI/OCR) | Team Member 2 (Focus: Frontends & UI Shells) |
| --- | --- |
| **Risk R-1 Mitigation:** Test Google Cloud Vision API on sample synthetic Latin/Bengali documents. Validate if preprocessing is needed. | **App Setup:** Initialize Flutter project. Set up Firebase for offline caching and basic app shell. |
| **Backend Setup:** Initialize FastAPI project (Python). Deploy skeleton to Render or Railway. | **Dashboard Setup:** Initialize Next.js/React project with Tailwind CSS. Deploy skeleton to Vercel. |
| **Mock APIs:** Create dummy JSON responses for `POST /diagnose` and `GET /clusters` as defined in section 5 of the SRS. | **API Mocks Integration:** Connect Flutter and Next.js shells to the mock JSON responses. |

---

## Phase 2: Core Logic & Citizen App Flow (Day 3-4)
**Goal:** Build the core diagnostic engine parsing user inputs, and build the citizen-facing mobile application flow.

| Team Member 1 (Focus: Matching & Rules) | Team Member 2 (Focus: Flutter App) |
| --- | --- |
| **Extraction & Matching:** Implement `indic-transliteration` and `rapidfuzz` logic to match OCR'd names across documents. | **Intake UI:** Build document upload screens, language selector, and symptom/location capture. |
| **Classification Rules:** Hardcode the 5-7 deterministic rules for root-cause classification. No LLMs on the decision path. | **Verification UI:** Build side-by-side verification view (extract real/scanned fields vs. what the system read). |
| **Explanation Layer:** Integrate Gemini API to generate plain-language explanations. Implement pre-generated local fallbacks. | **State Management:** Wire up the Flutter app to call the live internal `POST /diagnose` endpoint. |

---

## Phase 3: Event Store & Accountability Dashboard (Day 5-6)
**Goal:** Persist diagnosis events, compute clusters, and build the system accountability view for officials. 

| Team Member 1 (Focus: DB & Clustering) | Team Member 2 (Focus: Next.js Dashboard) |
| --- | --- |
| **Event Store:** Setup datastore (Postgres on Render or local SQLite). Write anonymized events generated by `/diagnose`. | **Dashboard Layout:** Build data tables and UI components for the web dashboard (Tailwind). |
| **Clustering Logic:** Implement `groupby` on event signatures (root cause + location) to form system defect clusters. | **Clusters View:** Present the ranked systemic defects list. Build expandable views to see underlying anonymized cases. |
| **Data Seeding:** Formulate synthetic, anonymized exclusion-event records to populate the datastore for demo scaling. | **Integration:** Connect the dashboard to the live `GET /clusters` API endpoint. |

---

## Phase 4: Integration, Stabilizing & Offline Fallbacks (Day 6-7)
**Goal:** Ensure the system works cohesively, handles network failures gracefully, and adheres to responsible AI requirements.

| Team Member 1 (Backend, AI Polish) | Team Member 2 (Frontend Polish, Error Handling) |
| --- | --- |
| **End-to-End Testing:** Run multi-document scenarios through `/diagnose` to verify OCR -> Match -> Rule -> Response flow. | **Feedback Loops:** Implement "Likely cause" disclaimers on Flutter UI. Add the flag/incorrect feedback button. |
| **Resilience & Fallbacks:** Test Gemini API failure pathways to ensure local explanation fallbacks fire correctly. | **Error states & Loading:** Add proper loading states during image upload/OCR processing, and bad-image upload prompts. |
| **Threshold Tuning:** Configure confidence thresholds for `GET /clusters` so low-frequency edge cases don't pollute the dashboard. | **Offline Testing:** Emulate network disconnects on the mobile app to verify shell resilience. |

---

## Phase 5: Demo Prep & Submission (Day 7-8)
**Goal:** Prepare the exact narrative and assets required for the Hackathon submission. The team works together here.

* **Collaborative Tasks:**
  * Define the demo "Golden Path": Use the "Rahima Begum" persona (English Aadhaar + Bengali Ration card -> Name Mismatch -> "You are one of many" dashboard zoom-out).
  * Record the screen-capture backup video in case of live environment failure.
  * Update `README.md` with setup instructions, free-tier tech stack disclosure, build-assistance tooling (Claude Code/DeepSeek/Stitch), and clear boundaries of what Mou *does not* do (i.e. changing government records).
  * Build the presentation deck on TWO pillars: (1) **systemic visibility** — the inversion (one rejection = misfortune; many sharing a root cause = an accountability report); and (2) **generality** — Mou is a *silent-exclusion engine for any government scheme*, proven live on PDS rations. Include **one mockup slide** applying the identical flow to a second scheme (e.g. a pension or scholarship name mismatch) — a visual mockup only, **no code**. Canonical narrative: `docs/positioning.md`.
